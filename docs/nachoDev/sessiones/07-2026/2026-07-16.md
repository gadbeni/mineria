## Trabajo 1 — Historial de modificaciones de usuario

**Commit:** `pendiente` — `feat(users): auditoría de creación y modificación de usuarios`

### Problema
Se pidió un historial que registre **qué se modifica en cada usuario** (crear, editar, activar/desactivar, cambio de contraseña), guardando en la base de datos un antes y un después (lo que cambió y lo que no), con una vista tipo línea de tiempo.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `database/migrations/..._create_user_edits_table.php` | tabla `user_edits` (before/after/changed/unchanged en JSON) |
| `app/Models/UserEdit.php` | modelo nuevo (casts array) |
| `app/Models/User.php` | `auditFields()`, `displayAudit()`, relación `edits()` |
| `app/Observers/UserObserver.php` | eventos `created` y `updated` que graban el historial |
| `app/Providers/AppServiceProvider.php` | registra el Observer y la acción de Voyager |
| `app/Http/Controllers/UserHistoryController.php` | muestra el historial de un usuario |
| `app/Actions/UserHistoryAction.php` | botón "Historial" en la lista de usuarios (Voyager) |
| `resources/views/users/history.blade.php` | vista timeline (cabecera + tarjetas Antes/Después) |
| `routes/web.php` | ruta `users.history` |

### Solución
- Se creó la tabla `user_edits` que guarda por cada acción: `before` y `after` (todos los campos auditados legibles), `changed` (solo lo modificado, con antes/después), `unchanged` (solo lo no modificado), más `action`, `edited_by` y `edited_at`.
- La captura se hace con un **Observer** (`created` y `updated`), no con el evento `booted()` del modelo (que no registraba el listener en este proyecto).
- Campos auditados: **Nombre, Email, Contraseña, Rol, Empresa, Estado, Avatar**. Los valores se guardan legibles (Rol → nombre del rol, Empresa → razón social, Estado → Activo/Inactivo).
- **Seguridad:** la contraseña **nunca se almacena ni se muestra**; solo se enmascara (`••••••••`) y se registra que cambió.
- Acciones legibles: `Usuario creado`, `Usuario modificado`, `Usuario activado`, `Usuario desactivado`, `Contraseña actualizada`.
- Vista **timeline** (como el diseño solicitado): cabecera con avatar/nombre/estado y conteo de registros; cada evento es una tarjeta con ícono/color por tipo y una tabla **Campo / Antes / Después** (resalta lo cambiado). En "Usuario creado" muestra "Datos iniciales del usuario" en lugar de "Se modificó N campos".
- Acceso: botón **"Historial"** en la lista de usuarios de Voyager, o `/admin/users/{id}/history`.

**Informe de trabajo — 16/07/2026**
**Responsable:** nachoDev

**Actividad:** Implementación de un historial de auditoría para los usuarios del sistema.

**Problema detectado:**
No existía un registro de los cambios realizados sobre los usuarios (creación, edición de datos, activación/desactivación o cambio de contraseña). No se podía saber quién modificó a un usuario, cuándo, ni qué datos cambiaron.

**Solución aplicada:**

Por seguridad, la contraseña se registra únicamente como "cambiada", sin guardar ni mostrar su valor.

**Detalle de archivos modificados:**
1. Base de datos: nueva tabla para el historial de usuarios.
2. Sistema: registro automático de creación y modificación de usuarios.
3. Pantalla: nueva vista de historial (línea de tiempo) con botón de acceso en la lista de usuarios.
4. Revisión del registro de peticiones al panel administrativo (middleware de log).

**Resultado:** Cada usuario cuenta ahora con un historial completo y seguro de sus cambios, visible desde una pantalla clara y ordenada.
